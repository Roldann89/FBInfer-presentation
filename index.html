01. Presentación

¿Qué es FBInfer?

Es un motor de análisis estático de código diseñado para encontrar fallos graves en el software antes de que sea compilado o enviado a producción.

  • Prevención de Crashes: Evita fallos críticos que detienen las aplicaciones (como punteros nulos o desbordamientos).
  • Análisis en Tiempo Real: Se integra en servidores de integración continua (CI) para validar cambios en minutos.
  • Multilenguaje: Soporte avanzado para ecosistemas diversos como Java, C, C++ y Objective-C.

¿Cómo opera el Análisis Estático?

A diferencia del análisis dinámico (pruebas unitarias/QA) que requiere ejecutar el software en escenarios de prueba, Infer analiza la estructura lógica y matemática del código fuente, razonando sobre todas las rutas de ejecución posibles simultáneamente.

02. Origen

Historia y Respaldo Empresarial

2000 - 2001

Lógica de Separación

John Reynolds, Peter O'Hearn e Ishtiaq inventan esta teoría matemática para razonar localmente sobre punteros de memoria, revolucionando la verificación formal.

2009

Startup Monoidics

Los creadores de la teoría fundan en Londres Monoidics para industrializar la lógica de separación en un analizador comercial llamado INFER.

2013

Adquisición de Facebook

Facebook adquiere Monoidics y a su equipo científico. El objetivo: solucionar los problemas de crashes en la escala gigante de sus apps de Android e iOS.

2015 - Hoy

Liberación Open Source

Meta libera Infer bajo licencia MIT. Actualmente es usado en producción por gigantes como **Spotify, Uber, Amazon, Mozilla y Riot Games**.

03. Innovación Teórica

Análisis Composicional y Bi-Abducción

El análisis estático clásico no escala porque debe evaluar millones de combinaciones globales. Infer revolucionó el campo con la **Bi-Abducción** (Bi-Abduction).

1
Análisis Local (Composicional)

Infer analiza cada función o método **por separado** sin mirar el resto del programa.

2
Cálculo de Pre y Postcondiciones

Genera una especificación matemática: qué necesita la función para no fallar (Pre) y qué altera al salir (Post).

3
Abducción de Faltantes

Al unir funciones, si falta información del estado, la abducción deduce qué trozo de memoria lógica falta para garantizar la seguridad.

Esquema de Bi-Abducción

Estado Actual HENO
Método requiere precondición PRE
Abducción: ¿Qué falta?
Deducción: [Falta MEMORIA X]
Resultado Garantizado POST

Gracias a este razonamiento incremental, Infer puede analizar miles de cambios de código en segundos de forma lineal.

03. Capacidades

Errores Críticos que Detecta

Infer no busca errores ortográficos o estéticos simples; busca fallos de memoria, concurrencia y recursos.

Punteros Nulos (Null Pointer)

Localiza desreferenciaciones accidentales a objetos nulos que provocan caídas inmediatas de la aplicación (como NullPointerException).

JavaC/C++

Fugas de Memoria (Memory Leaks)

Encuentra asignaciones de memoria dinámica en el montón (heap) mediante malloc que nunca se liberan con free.

C/C++Obj-C

Fugas de Recursos (Resource Leaks)

Detecta conexiones a bases de datos, sockets de red y flujos de archivos que no se cierran al ocurrir una excepción.

JavaC/C++

Concurrencia (Thread Safety)

Analiza anotaciones de seguridad de hilos y detecta condiciones de carrera (Data Races) y bloqueos mutuos (Deadlocks).

04. Instalación

Facilidad de Instalación por Plataforma

Infer está diseñado nativamente para sistemas basados en Unix (Linux y macOS). En Windows requiere alternativas de virtualización.

macOS

Extremadamente Fácil

Se instala automáticamente en segundos utilizando el gestor de paquetes Homebrew.

brew install infer

Linux

Fácil / Estándar

Descargando las versiones precompiladas (.tar.xz) desde GitHub y agregándolas al PATH.

curl -sSL https://github.com/... | tar -xJ
export PATH="/opt/infer/...:$PATH"

Windows

Requiere Adaptación

No hay soporte nativo directo. Compilarlo en Windows nativo es inviable por su dependencia estricta de OCaml y Clang Unix.

Alternativas recomendadas:
  • WSL2 (Windows Subsystem for Linux)
  • Contenedores Docker (Aislamiento completo)

06. Demo Práctica

Ejemplo de Uso y Reporte de Bugs

Infer intercepta el compilador e identifica errores complejos. Haz clic en las pestañas para ver los casos probados.

Ejemplo.java
import java.io.*;

public class Ejemplo {
    public int obtenerLongitud(String texto) {
        String miTexto = null;
        if (texto != null) miTexto = texto;
        
        // BUG 1: Puntero Nulo (puede explotar)
        return miTexto.length(); 
    }

    public void escribir(String ruta, String d) throws Exception {
        // BUG 2: Fuga de Recurso (FileOutputStream)
        FileOutputStream fos = new FileOutputStream(ruta);
        fos.write(d.getBytes());
        fos.close(); 
    }
}
Comando y Salida de Infer

leak.c
#include <stdlib.h>
#include <stdio.h>

void procesar(int cond) {
    // Se reserva memoria en el heap
    int *buffer = (int *)malloc(100 * sizeof(int));
    
    if (cond > 50) {
        printf("Error fatal\n");
        // BUG: Salimos sin liberar con free()
        return; 
    }
    
    free(buffer); // Proceso normal
}

07. Resumen

Conclusiones del Análisis de FBInfer

Evaluación final sobre el impacto de la herramienta en la ingeniería de software actual.

Puntos Fuertes (Ventajas)

  • Matemáticamente infalible: La lógica de separación y bi-abducción le otorgan una tasa bajísima de falsos positivos frente a herramientas basadas puramente en patrones sintácticos.
  • Escalabilidad Real: Es capaz de analizar proyectos inmensos gracias a su carácter composicional (métodos analizados por separado).
  • Integración en CI/CD: Se ejecuta sobre la marcha en commits y pull requests dando un feedback inmediato a los programadores.

Limitaciones (Inconvenientes)

  • Sin soporte directo para Windows: Requiere WSL2 o contenedores Docker, complicando ligeramente la configuración nativa.
  • Curva de Aprendizaje: Algunos reportes avanzados de concurrencia y flujo lógico requieren formación técnica sólida sobre gestión de memoria.
  • Dependencia de Compiladores: Requiere interceptar la compilación, lo que a veces complica el setup en proyectos con sistemas de compilación no estándar.
